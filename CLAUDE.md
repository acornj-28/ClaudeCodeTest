## Running the games

No build step — open HTML files directly in a browser:

```
game/index.html       # Gungeon Zero (modular source, multi-file)
gungeon-zero.html     # Gungeon Zero (single self-contained file)
tictactoe.html        # Tic Tac Toe
```

To launch from the terminal: `start <filename>.html` (Windows) or `open <filename>.html` (Mac).

## Architecture: Gungeon Zero

The game has two forms that must be kept in sync:
- **`game/`** — modular source split across ~19 files (the working copy)
- **`gungeon-zero.html`** — all JS inlined into one file (for sharing/distribution)

When modifying the game, edit the modular source in `game/` first, then regenerate `gungeon-zero.html` by inlining all scripts in the correct load order.

### Script load order (defined in `game/index.html`)

Dependencies must load before dependents. The required order is:

```
constants.js → utils.js → input.js →
particle.js → bullet.js → enemy.js → player.js →
spawner.js → collision.js →
drawEffects.js → drawEnemy.js → drawPlayer.js →
hud.js → renderer.js →
menu.js → levelComplete.js → gameOver.js →
state.js → main.js
```

All files use **global scope** (no ES modules) — this is intentional to avoid `file://` CORS issues.

### State machine

`state.js` owns the game state machine. The flow is:

```
MENU → PLAYING → WAVE_TRANSITION → PLAYING → ... → LEVEL_COMPLETE → PLAYING (next level)
                                                                   ↘ GAME_OVER → MENU
```

`stateManager` holds a `gameContext` (gc) object with: `player`, `enemies[]`, `bullets[]`, `particles` (ParticleSystem), `score`, `level`, `wave`.

### Entity update pattern

Each entity owns its own `update(dt)`. Systems (`collision.js`, `spawner.js`) operate across entities. The PLAYING state in `state.js` orchestrates the update order: player → enemies → bullets → particles → collision → cleanup → wave check.

### Enemy system

All enemies extend `Enemy`. Each subclass overrides `setupStats(level)` and `behaviorUpdate(dt, player)`. Stats are automatically scaled per level via `scaleStats()`. New enemy types need: a class in `enemy.js`, a draw function in `drawEnemy.js`, and a case in `createEnemy()`.

### Sprites

All drawing uses Canvas 2D primitives only — no image files. Drawing functions are pure functions in `sprites/` that take `(ctx, entity)`. `drawEnemy(ctx, enemy)` dispatches by `enemy.type` string.

### Level data

Levels 1–3 are hand-defined in `LEVELS[]` in `constants.js`. Level 4+ are procedurally generated by `generateLevel(n)`. `getLevel(n)` handles the dispatch. To tweak difficulty, edit the wave objects: `{ grunts, rushers, tanks, spawnInterval }`.
