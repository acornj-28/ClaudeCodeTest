## Project overview

This repo contains two browser games built from scratch with pure HTML5 Canvas + vanilla JavaScript — no frameworks, no build tools, no dependencies.

**Gungeon Zero** — a retro top-down shooter. The player moves with WASD/arrow keys, aims with the mouse, shoots with left click, and pauses with Escape. Enemies spawn from the screen edges in waves and converge on the player. Three enemy types: Grunt (direct charge), Rusher (charge/pause cycle with speed trail), Tank (slow, heavy, periodic stomp burst with HP bar). Killing enemies scores points. Waves escalate within levels; completing all waves in a level triggers a level complete screen and the next level begins with scaled-up enemies. Game over on death, with restart or menu options.

**Tic Tac Toe** — simple 2-player browser game with score tracking across games (`tictactoe.html`).

## Running the games

No build step — open HTML files directly in a browser:

```
game/index.html       # Gungeon Zero (modular source, multi-file)
gungeon-zero.html     # Gungeon Zero (single self-contained file, for sharing)
tictactoe.html        # Tic Tac Toe
```

To launch from the terminal: `start <filename>.html` (Windows) or `open <filename>.html` (Mac).

## Dual-file rule (important)

Gungeon Zero exists in two forms that **must always be kept in sync**:
- **`game/`** — modular source split across ~20 files (the working copy to edit)
- **`gungeon-zero.html`** — all JS inlined into one `<script>` block (for sharing/distribution)

Always edit `game/` first. After changes are working, manually apply the same changes to the corresponding section in `gungeon-zero.html`. Sections in that file are marked with banner comments like `// ─── constants.js ───`.

## Architecture: Gungeon Zero

### Script load order (defined in `game/index.html`)

Dependencies must load before dependents. The required order is:

```
constants.js → utils.js → input.js →
particle.js → bullet.js → enemy.js → player.js →
spawner.js → collision.js →
drawEffects.js → drawEnemy.js → drawPlayer.js →
hud.js → renderer.js →
menu.js → paused.js → levelComplete.js → gameOver.js →
state.js → main.js
```

All files use **global scope** (no ES modules) — this is intentional to avoid `file://` CORS issues.

### State machine

`state.js` owns the game state machine. The full flow is:

```
MENU → PLAYING ⇄ PAUSED
         ↓
   WAVE_TRANSITION
         ↓
      PLAYING
         ↓ (all waves done)
   LEVEL_COMPLETE → PLAYING (next level, player reset)
         ↓ (player dies)
      GAME_OVER → MENU or PLAYING (restart)
```

`stateManager` holds a `gameContext` (gc) object with: `player`, `enemies[]`, `bullets[]`, `particles` (ParticleSystem), `score`, `level`, `wave`.

### Entity update pattern

Each entity owns its own `update(dt)`. Systems (`collision.js`, `spawner.js`) operate across entities. The PLAYING state in `state.js` orchestrates the update order: player → enemies → bullets → particles → collision → cleanup → wave check.

### Enemy system

All enemies extend `Enemy`. Each subclass overrides `setupStats(level)` and `behaviorUpdate(dt, player)`. Stats are automatically scaled per level via `scaleStats()`. New enemy types need: a class in `enemy.js`, a draw function in `drawEnemy.js`, and a case in `createEnemy()`.

### Sprites

All drawing uses Canvas 2D primitives only — no image files. Drawing functions are pure functions in `sprites/` that take `(ctx, entity)`. `drawEnemy(ctx, enemy)` dispatches by `enemy.type` string.

### Level data

Levels 1–3 are hand-defined in `LEVELS[]` in `constants.js`. Level 4+ are procedurally generated by `generateLevel(n)`. `getLevel(n)` handles the dispatch. To tweak difficulty, edit the wave objects: `{ grunts, rushers, tanks, spawnInterval }`. Enemy HP and speed scale by +15% and +5% per level respectively via `scaleStats()`.
